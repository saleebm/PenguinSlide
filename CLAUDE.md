## Project shape

Tilt-controlled iOS game (iOS 17+, Swift 5, SpriteKit + SwiftUI). A penguin slides on ice dodging falling icicles. Bundle id `dev.copt.PenguinSlide`. The Xcode project is **generated by xcodegen** from `project.yml` — that file is the source of truth, not `PenguinSlide.xcodeproj/project.pbxproj`. After editing `project.yml`, run `xcodegen generate` (or pass `REGEN=1` to any run/test script).

## Build, run, test

All scripts live at the repo root, use `set -euo pipefail`, and take env-var overrides. They build in **target mode** (`-target` + `-sdk`), *not* `-scheme` — scheme-based destination resolution silently fails under Xcode 26 platform churn ("Supported platforms ... is empty").

```sh
./run-sim.sh                 # build + install + launch on iOS Simulator
./run-device.sh              # build + install + launch on a physical iPhone (devicectl)
REGEN=1 ./run-sim.sh         # re-run xcodegen first (after editing project.yml)

./test-xcui.sh               # XCUITest target via `xcodebuild test`
TEST=PenguinSlideUITests/PenguinSlideUITests/testTitleScreenAppears ./test-xcui.sh   # single test
./test-smoke.sh              # agent-device UI loop → screenshots in test-evidence/
./test-perf.sh               # agent-device perf capture → test-evidence/perf.json
BUILD=1 ./test-smoke.sh      # chain run-sim.sh first

./scripts/inject-tilt.sh 0.7 # feed simulated tilt to a DEBUG sim build (see below)
./scripts/bump-version.sh patch|minor|major|build   # edits Info.plist
./scripts/archive.sh         # Release archive + signed .ipa → build/  (then scripts/upload.sh)
```

Full testing detail (agent-device version requirement, simulator perf gaps, the two iPhone 17 Pro UDIDs gotcha) is in [TESTING.md](TESTING.md). Real gameplay validation needs a **physical device** — the simulator has no gyro.

### Forcing game-over in tests

`GameScene` installs a hidden `SKLabelNode` named/labelled `debugForceGameOver` under `#if DEBUG`. Tapping it routes straight through `triggerGameOver()`, so tests never depend on physics timing. agent-device: `agent-device press 'label="debugForceGameOver"'`; XCUITest: `app.otherElements["debugForceGameOver"].tap()`. It must be an `SKLabelNode` — SpriteKit only surfaces label nodes to XCUITest's accessibility traversal. "Element not found" usually means a Release build (node excluded). Tests also target SpriteKit nodes by their accessibility `label` (e.g. `"Tap to start"`); changing that copy means updating the tests in the same change.

## Architecture

**SwiftUI is a thin shell around a SpriteKit scene.** `ContentView` holds a single `GameScene` in `@State` (built once in `.onAppear` so redraws don't recreate it) and renders it via `SpriteView`. Overlays — settings and game-over — are SwiftUI layered on top; gameplay HUD (score/best/hearts) is drawn inside the scene by `HUDController`.

`GameScene` is the orchestrator. `didMove(to:)` builds the static world (sky backdrop, ice field) and three subsystems, then routes per-frame updates, tilt input, and physics contacts:

- **`Penguin`** — player sprite + physics body + tilt-driven motion/lean, plus HP / i-frames / knockback. Exposes `tryTakeHit(from:)`, `isAlive()`, an `onHealthChanged` callback, and `reset()`.
- **`IcicleSystem`** — icicle lifecycle: spawn cadence, warning telegraph, per-icicle gravity, landing detection, shatter shards, camera shake, and impact audio/haptics. Owns `onIcicleHitPenguin(...)`.
- **`HUDController`** — pure presentation (score, best, hearts, start prompt, death flash). Reads no game state; `GameScene` pushes updates via its `setScore`/`setBest`/`setHealth` API.

Round end flows out of the scene: `triggerGameOver()` freezes the world (`physicsWorld.speed = 0` + pausing shard SKActions) and fires `onGameOver(GameResult)`. `ContentView` presents `GameOverView` (SwiftUI) over the frozen scene; its "Play Again" button calls back into `GameScene.playAgain()` → `restart()`.

### Physics is manual, by design

- Scene gravity is **zero**. The penguin's body is dynamic but `collisionBitMask = 0`; its position is written manually every frame (single source of truth — SpriteKit must not displace it), and knockback is applied directly to `Penguin.vx`. Icicles integrate their own per-node gravity in `IcicleSystem` because `SKPhysicsBody` has no per-body gravity scale. Physics bodies exist almost solely for **contact detection** (`Category` bitmasks in `Tuning.swift`).
- **Update-loop ordering contract** (documented in `GameScene.update(_:)`): all per-frame physics work must stay behind the `isStarted && !isGameOver` guard. `didBegin(_:)` runs *after* `update(_:)` within the same frame, so the death frame relies on the next frame's guard + `physicsWorld.speed = 0` to halt motion. Moving integration calls outside the guard yields one frame of post-death physics (teleporting shards).
- **`dt == 0` sentinel.** Setting `lastUpdateTime = 0` makes the next frame compute `dt = 0`. This is reset on start, restart, settings-resume, and app-foreground so a multi-second pause (screen lock, settings sheet) doesn't produce a giant `dt` that teleports the penguin / balloons the score. Keep this symmetric across all resume paths.

### Tilt input

`GameScene.currentTilt()` reads `CMMotionManager`'s fused `deviceMotion.gravity` synchronously in `update(_:)` (no callback queue — frame-locked, no main-queue storm). `screenGravity(_:)` rotates device-frame gravity into interface space so "tilt-right is positive" holds for every interface orientation. Fallbacks, in priority order: `MotionInjector` (DEBUG + simulator only, a `Network.framework` listener on `127.0.0.1:7654` fed by `scripts/inject-tilt.sh`) → `GCKeyboard` arrows / A,D (simulator). A small dead zone suppresses flat-phone drift.

### Tuning

`Tuning.swift` is the difficulty hub, grouped by subsystem (`Icicle`, `Chase`, `Feel`, `Run`) plus the `Category` bitmasks. The README's "Where to tweak difficulty" table maps each knob to its effect. `Tuning.Penguin` is special: it's a **mutable `PenguinTuning` struct** (`PenguinTuning.swift`) loaded from UserDefaults on first access, so the settings UI / debug menu can override input-feel knobs at runtime. Call sites still read `Tuning.Penguin.maxSpeed` etc.

### Assets & persistence

- **`SpriteCatalog`** is the single entry point for art from `Assets.xcassets` (cached `SKTexture`s, nearest-neighbor filtering for crisp pixel art, plus a `tiled(...)` helper). Adding a sprite = one `Sprite` enum case + one imageset. **`PenguinAnimations`** slices the SpriteCook-generated horizontal spritesheets into per-state frame arrays (`spritecook-assets.json` records the source asset IDs / prompts / frame counts).
- Persistence is all on-device UserDefaults, no accounts/network: `best_score` (auto-written by `GameScene` each round), `HighScores` (a JSON list, only appended when the player deliberately saves a named run on the game-over page), and `PlayerProfile.name`.

## Conventions

- Sim-only and test-only code is gated behind `#if DEBUG` (and `targetEnvironment(simulator)` where relevant) so Release/device builds are unaffected.
- Audio: app sets `AVAudioSession` `.playback + .mixWithOthers` (plays through the silent switch, mixes under the user's music). `SKAudioNode` runs on the audio engine independent of the scene clock, so background music is pause/resumed explicitly on app-lifecycle and settings transitions.
- Motion permission (`NSMotionUsageDescription`) is required in `Info.plist` or CoreMotion crashes on first use. `GENERATE_INFOPLIST_FILE = NO`; the checked-in `Info.plist` is authoritative.
